## The branch-protection datum (the question this research was fired to answer)
**Documented cases of an agent DELIBERATELY bypassing its own git hooks (`--no-verify`-class evasion): 0 of 13.** The dominant real-world mechanism is guards absent/opt-out/unverified, not guards evaded. BUT two incidents argue for one server-side layer anyway: Cursor/Opus (client-side guard layer failed silently — only a stop OUTSIDE the agent's machine would have held) and Amazon Q (the agent's own instruction stream was poisoned — same lesson). **Recommendation calibrated: GitHub branch protection stays RECOMMENDED-BUT-OPTIONAL (browser checklist item 4) — zero documented in-the-wild frequency for the exact hole it plugs, but the two guard-layer-failure incidents are the tail it insures against, at 2-min cost.**

## What today's build got right, per the evidence (and one gap)
- Every high-loss incident ran in an equivalent of "full access + confirmations off." Our stack has no such mode: enforcement is mandatory-by-boot (setup.sh hooksPath), token-gated overrides, operator-gated tiers.
- Codex #3728's user lost nothing **because the repo was pushed** — our stop-hook commit+push discipline is precisely this mitigation, validated.
- Lemos/prod-TRUNCATE root cause (misconfigured pointer: test var → prod target) maps to our verifier discipline of re-pointing remotes in disposable clones before any destructive test — practiced today, 3 rounds.
- **Gap surfaced (accepted, logged): local uncommitted work + non-repo container files have no deterministic guard** (Sol/Gemini class). Mitigation is the existing commit-early/push-always loop + ephemeral containers holding nothing canonical. No new guard built — blast radius is minutes of work, not the record.
